\section{Input and Output Structure}

The user-facing input path is the Tkinter desktop workflow. A student can start from a blank model or a two-dimensional shear-frame template, define nodes and members on the canvas, assign material and section properties, apply supports, add nodal or member loads, assign lumped masses for modal analysis, validate the model, and then run Static or Modal analysis. The property panel and object tree are used to inspect and revise selected model objects without editing source code or writing XML manually.

XML is the reproducibility path. A desktop-created model can be saved or exported to XML, and a saved XML file can be loaded back into the same \codeterm{ModelBuilder} and \codeterm{StructuralModel} pipeline. The XML file records analysis-relevant model information including nodes, elements, materials, sections, supports, releases, loads, support settlements, masses, diaphragm groups where present, unit system, and load cases. XML is therefore a backend interchange and archive format, not a requirement placed on ordinary users.

Static output includes the DOF map, full stiffness matrix \(K\), reduced stiffness matrix \(K_{ff}\), global force vector \(F\), reduced force vector \(F_f\), nodal displacements, support reactions, local member-end forces, and axial/shear/moment diagram data. Modal output includes stiffness and mass matrices, reduced or condensed dynamic matrices where available, eigenvalues, circular frequencies, ordinary frequencies, periods, mode shapes, modal masses, participation factors, effective modal masses, and mass participation ratios.

Visible result tables and plots can be exported where implemented. XML model export remains the primary persistence format for reproducing a complete model, while text, CSV, and PNG exports support classroom reporting and quick review of the currently visible results.
