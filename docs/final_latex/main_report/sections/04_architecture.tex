\section{Software Architecture and Object-Oriented Design}

The architecture follows one object-oriented pipeline:
\begin{center}
\begin{tabular}{c}
Tkinter canvas/tables/templates/XML \\
\(\rightarrow\) \codeterm{ModelBuilder}
\(\rightarrow\) \codeterm{StructuralModel}
\(\rightarrow\) DOF/K/M/F assembly \\
\(\rightarrow\) Static or Modal solver
\(\rightarrow\) result object
\(\rightarrow\) tables/plots/export
\end{tabular}
\end{center}

This design keeps responsibilities separated. \codeterm{ModelBuilder} creates models from programmatic calls, templates, XML, tables, and desktop actions. \codeterm{StructuralModel} stores nodes, elements, materials, sections, supports, loads, masses, releases, and validation state; it does not solve or plot. \codeterm{StructuralValidator} centralizes validity checks. DOF and assembly modules create \(K\), \(M\), \(F\), and reduced systems. Static and Modal solvers operate on assembled matrices and return result objects. UI and controller code coordinate user actions and display results without owning solver math.

Frames, trusses, shear frames, cantilevers, and benchmark problems all become ordinary model instances before assembly. This avoids duplicate solver logic by model family and makes validation stronger because each model type exercises the same Static and Modal computational path.
