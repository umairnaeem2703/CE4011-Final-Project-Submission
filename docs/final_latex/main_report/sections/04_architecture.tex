\section{Software Architecture and Object-Oriented Design}

The software is organized around a single object-oriented pipeline:
\begin{center}
\begin{tabular}{c}
Tkinter canvas/tables/templates/XML \\
\(\rightarrow\) \codeterm{ModelBuilder}
\(\rightarrow\) \codeterm{StructuralModel}
\(\rightarrow\) DOF/K/M/F assembly \\
\(\rightarrow\) Static or Modal solver
\(\rightarrow\) result object
\(\rightarrow\) tables/plots/export
\end{tabular}
\end{center}

This pipeline is the main design decision of the project. UI actions create or modify model data; model data is validated; assembly modules create numerical systems; solvers operate on those systems; result objects preserve outputs; and viewers/exporters consume result objects. Solver math is therefore not duplicated in Tkinter widgets or controller methods.

\codeterm{ModelBuilder} is the primary creation API. It is used by programmatic examples, templates, XML-backed workflows, and desktop interactions so that a model created through the GUI follows the same path as a model created for a regression test. \codeterm{StructuralModel} stores data and validation state; it does not solve or plot. \codeterm{StructuralValidator} is reused for model validation rather than reimplementing structural checks in each UI command.

The solver layer is intentionally generalized. A shear-frame template, a truss example, a cantilever, and a SAP2000 benchmark all become ordinary model instances before assembly. This is a practical object-oriented design choice: it avoids duplicate static/modal logic by model family and makes verification stronger because multiple structure types exercise the same numerical engine. XML remains the reproducibility backend, while the desktop workflow remains the user-facing model creation and result review path.
