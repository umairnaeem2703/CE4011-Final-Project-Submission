\section{Software Architecture and Object-Oriented Design}

The architecture follows a single analysis pipeline:
\begin{center}
\begin{tabular}{c}
Tkinter canvas/templates/XML \\
\(\rightarrow\) \codeterm{ModelBuilder}
\(\rightarrow\) \codeterm{StructuralModel}
\(\rightarrow\) DOF/K/M/F assembly \\
\(\rightarrow\) Static or Modal solver
\(\rightarrow\) result object
\(\rightarrow\) plots/tables/export
\end{tabular}
\end{center}

Component & Responsibility \\
\midrule
\codeterm{ModelBuilder} & Creates models from programmatic calls, templates, tables, XML, and graphical input. \\
\codeterm{StructuralModel} & Stores nodes, elements, supports, properties, loads, masses, releases, and validation state. It does not solve or plot. \\
\codeterm{StructuralValidator} & Centralizes model validation checks so UI code does not duplicate structural rules. \\
Assembly modules & Create DOF maps, stiffness matrices, force vectors, mass matrices, and reduced systems. \\
Static and Modal solvers & Operate on assembled matrices/vectors and return traceable result objects. \\
\codeterm{Controller} & Coordinates user actions without owning solver math or Tkinter-specific drawing logic. \\
Visualization/export & Consume models and result objects to create tables, plots, XML exports, text/CSV exports, and PNG plot exports. \\
\bottomrule
\end{tabular}

The solver does not branch by structure family. A shear frame template, a truss model, a cantilever, and a SAP2000 benchmark all become ordinary instances of the same model representation before assembly. This design keeps the computational engine reusable and makes verification tests more meaningful because every model family exercises the same numerical path.

XML is the reproducibility and interchange backend. Users are not expected to hand-write XML; the desktop workflow creates, saves, loads, and exports models through the same backend representation.
